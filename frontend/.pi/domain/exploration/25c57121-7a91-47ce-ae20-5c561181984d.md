## Business Context

Keystone is an **OpenAPI Specification Governance Server**. It ingests OpenAPI specs from repositories, detects breaking changes between spec versions, evaluates specs against configurable policy rules (written in a Policy DSL), tracks service dependencies in a graph for impact analysis, provides a governance health dashboard, and dispatches notifications via multiple channels (Slack, Email, Webhook).

The frontend is a **Next.js (App Router) dashboard** with a print-newspaper design aesthetic. It surfaces governance data across six views: Overview, API Inventory, Breaking Changes, Policy Compliance, Dependency Graph, and Notifications.

## Assumptions

| Assumption | Impact if Wrong | Mitigation |
|------------|----------------|-----------|
| The backend API is deployed and accessible at a configurable URL | Frontend cannot fetch data; must mock or fall back | Use `NEXT_PUBLIC_KEYSTONE_API_URL` env var; implement graceful error states and skeleton loading |
| Backend API response shapes match the defined DTOs (snake_case JSON fields) | Frontend deserialization breaks; mismatched fields | Define TypeScript interfaces matching backend DTOs; add runtime validation layer |
| Auth is handled by the backend (RBAC) and the frontend only needs to pass tokens | Frontend cannot secure admin-only actions without backend enforcement | Pass Bearer tokens in headers; conditionally show/hide UI elements based on role |
| The frontend is a single-page app with searchParam-based routing | URL structure may need to change for SEO or deep linking | Keep view state in `?view=` search param; use Next.js App Router for flexibility |
| Policy DSL expressions are stored and evaluated server-side only | No policy evaluation logic exists in the frontend | Frontend only displays policy metadata and evaluation results; no DSL parsing |
| The dependency graph is pre-computed server-side; frontend only renders static SVG data | Frontend needs to compute layout if backend only sends raw graph data | Backend returns positioned nodes (x,y); fall back to force-directed layout if missing |
| The dashboard is desktop-first (min 1024px) | Mobile users have degraded experience | Communicate desktop-first constraint; ensure horizontal scroll on tables |
| Breaking change reports include raw diff text with `-`/`+` markers for rendering | Diff format may differ or be unavailable | Render diff with simple line-prefix detection; add fallback display |
| Notifications are pulled via REST API (polling), not WebSocket/push | Stale notification state between polls | Implement periodic refetch (30s interval) for unread count |
| Theme toggle is purely CSS-based using `data-theme` attribute on `<html>` | Theme changes cause layout flash or FOUC | Apply theme before hydration via inline script in `<head>` |
| The frontend does not need to handle spec file uploads directly | Mismatch if CLI/API workflow changes | UI only displays ingestion results; CLI/CI handles uploads |
