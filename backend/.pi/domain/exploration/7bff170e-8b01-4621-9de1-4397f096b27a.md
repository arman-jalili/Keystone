## Business Context

We are building Keystone, an enterprise API contract governance platform. Detects breaking changes in OpenAPI specs before production. Modules: Contract Ingestion, Breaking Change Analysis, Policy Engine, Notification Engine, Dependency Graph, Dashboard, CLI Orchestrator. Users: Platform engineers, API owners, compliance managers. Constraints: Local CLI analysis must complete within 50ms (in-CI-runner); remote audit upload is async (<1s); must integrate with GitHub/GitLab status APIs.

## Assumptions

| Assumption | Impact if Wrong | Mitigation |
|------------|----------------|-----------|
| OpenAPI specs are the single source of truth for API contracts | If teams use other contract formats (gRPC proto, GraphQL SDL, AsyncAPI), analysis will miss changes | Design analysis engine with a pluggable contract parser interface; support OpenAPI as first parser, extend later |
| CI pipelines have network access to Keystone services | If CI runners run in isolated networks with no egress, the status API and CLI cannot reach Keystone | Provide a local CLI mode that outputs results as files; support air-gapped deployment via pull-based artifact upload |
| Breaking changes are detected by diffing two sequential spec versions | If teams skip versions or squash commits, the diff base is lost | Store every ingested spec version; support explicit base reference (e.g., git ref, tag, or commit SHA) |
| Policy rules are authored by compliance managers only | If API owners also need to define policies, role model needs expansion | Design Policy as a tenant-scoped resource with RBAC; allow delegation |
| GitHub/GitLab API is always available | If VCS APIs are down, status reports fail silently | Implement retry with exponential backoff + circuit breaker; queue failed notifications for replay |
| The <10ms overhead assumption was incorrect; actual target is 50ms local, 1s async remote | Architecture was designed around a remote server call that cannot meet <10ms | Separate local CLI analysis (fast, cached) from remote audit upload (async, fire-and-forget) |
| Dependency graph must be discoverable from OpenAPI specs alone | OpenAPI specs only declare producer contracts, not consumer relationships | For v1, support explicit dependency declarations via `keystone.yml`; defer automated consumer discovery to a future Service Discovery module |
| CI pipelines always provide a clean sequential spec history | Squashed commits, force pushes, and branch rebasing can skip spec versions | Support three-way fallback for diff base: explicit ref > previous ingested version > latest version on main branch |

## Key Decisions

1. **Policy DSL + UI (both)** — DSL (YAML) is the source of truth for GitOps compatibility; the UI edits the DSL and commits changes to a dedicated repo via GitHub API. Implements a `PolicyRepository` service that can pull from git or accept UI mutations.
2. **Multi-environment with inheritance** — Each Policy has an environment label (dev/staging/prod or * for all). The CI call includes the target environment. Dashboard shows per-environment compliance.
3. **Three-layer spec version resolution** — (1) Explicit base ref from CI, (2) previous ingested version of same spec, (3) latest version on main branch. All three supported; diff engine accepts optional `base_ref`.
4. **Pluggable PolicySource interface** — v1 supports local file + HTTP(s) fetch; later add OCI registry for organization-wide policy hubs.
5. **Synchronous local verdict + async notifications** — The CLI performs fast local analysis (blocking, returns verdict immediately) and asynchronously uploads to the server for audit persistence. No blocking on remote calls.
